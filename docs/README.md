# Документация Brai New

Это входная точка в документацию проекта. Здесь описываются система, её
технологический стек, инженерные процедуры и причины архитектурных решений.
Нормативные требования не копируются сюда: их источниками остаются
[`AGENTS.md`](../AGENTS.md), активные OpenSpec-спецификации и текущий код с
тестами.

## Как устроена документация

Документация ведётся по **Diátaxis**. Подробный проектный стандарт находится в
[`documentation-methodology.md`](documentation-methodology.md). Методология
разделяет материалы по намерению читателя, поэтому один файл не должен
одновременно быть учебником, справочником и журналом решений.

| Раздел                                    | Вопрос читателя                               | Что туда попадает                                      |
| ----------------------------------------- | --------------------------------------------- | ------------------------------------------------------ |
| [`tutorials/`](tutorials/README.md)       | «Покажи мне путь от нуля до результата»       | Пошаговое обучение с рабочим результатом               |
| [`how-to/`](how-to/README.md)             | «Как выполнить конкретную задачу?»            | Процедуры и runbook-ы                                  |
| [`reference/`](reference/README.md)       | «Как устроено и какие есть точные параметры?» | Команды, карты, контракты и инвентари                  |
| [`explanation/`](explanation/README.md)   | «Почему система устроена именно так?»         | Архитектура, границы и trade-off-ы                     |
| [`architecture/`](architecture/README.md) | «Где проходят системные границы?»             | Навигация по архитектуре и access/isolation материалам |
| [`stack/`](stack/README.md)               | «Какие технологии мы используем?»             | Версии, назначение, владельцы и источники истины       |
| [`decisions/`](decisions/README.md)       | «Какое решение приняли и почему?»             | ADR, которые фиксируют долгоживущие решения            |

Подробные текущие документы по уже реализованной модели доступа сохранены на
верхнем уровне для обратной совместимости:

- [Архитектура прав и изоляции](agent-access-architecture.md)
- [Права, пользовательские среды и дисковые квоты](permissions-and-isolation.md)

ADR Brai New хранятся в [`decisions/`](decisions/README.md), собираются
локальным Log4brains в тёмной теме Brai и автоматически публикуются после
проверок на защищённом [`adr.brai.one`](https://adr.brai.one/). Старые
ADR-записи проекта `brai` в новый каталог не импортируются.

Если найдено расхождение, сначала исправляется источник с более высоким
приоритетом, затем краткая документация и Memory Bank. Версии и команды нельзя
«исправлять» только в Markdown.

## Правила обновления

Агенты проходят единый project-local workflow `docflow`: он автоматически
выбирает глубину `quick`/`normal`/`full`, проверяет затронутые источники и не
считает задачу закрытой при нерешённом governance. Пользователь не обязан
вводить внутренние OpenSpec или `docflow` команды; точка входа для агента —
корневой [`AGENTS.md`](../AGENTS.md) и skill `docflow`.

1. Сначала определить тип материала по Diátaxis.
2. Проверить источник истины: код, тест, OpenSpec, манифест или operator README.
3. Обновить существующий документ, а не создавать дубликат с новым названием.
4. Для архитектурного решения добавить ADR или ссылку на существующий ADR.
5. Для изменения доступа, границ сервисов или security-контракта синхронно
   обновить OpenSpec и активную задачу согласно `AGENTS.md`.
6. Для изменения installed tooling или host-сервиса обновить
   `/home/mark/DEPLOYMENT.md` в той же операции. Секреты туда не записываются.
7. Проверить ссылки, Markdown и команды, которые были изменены.

## Статусы документов

- `draft` — заготовка, которую нельзя считать контрактом;
- `active` — текущая поддерживаемая версия;
- `deprecated` — документ оставлен для перехода, но не расширяется;
- `archived` — исторический материал, не использовать как источник решения.

Статус пишется в начале документа и не заменяет проверку нормативного источника.
