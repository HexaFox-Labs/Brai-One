# Как добавить инструмент в стек

**Статус:** `active`

Этот маршрут описывает внутреннюю работу агента после установки, обновления или
удаления инструмента. Пользователь не запускает отдельную stack-команду и не
заполняет generated pages вручную: агент завершает регистрацию в той же задаче.
В результате одна запись manifest обновляет страницу инструмента, индекс
категории и JSON-каталог для будущего сайта.

## 1. Проверь фактическую установку

Найди источник истины: package manifest, конфигурацию, installer или
`/home/mark/DEPLOYMENT.md` для host-level инструмента. Не записывай в каталог
пароли, токены, приватные ключи или содержимое secret-файлов.

Для RTK факты выглядят так:

- версия: `0.42.4`;
- установка: `/srv/opt/rtk/bin/rtk`;
- проверка: `rtk --version` и `which rtk`;
- правила использования: [`RTK.md`](../../RTK.md).

## 2. Добавь одну запись

Отредактируй [`tools/stack/catalog.json`](../../tools/stack/catalog.json). В
записи обязательны стабильный `id`, название, категория, тип, область, краткое
описание, объяснение «что это», назначение в Brai, способ использования,
статус, версия, location, source references и verification.
Дополнительно обязательны подробные поля `details`: почему выбран инструмент,
как он работает именно в нашем контуре, его возможности, практические сценарии,
типичные ошибки и связанные инструменты. Укажи минимум две возможности и два
сценария; пиши их конкретно, а не рекламными словами. Главные поля страницы —
`details.whatItIsDetailed` и `details.whyNeededDetailed`: каждое должно быть
самостоятельным объяснением минимум из двух предложений и примерно 180+
символов. Первое объясняет, что это за класс инструмента и каков его основной
механизм; второе — какую проблему Brai он решает, какую пользу даёт и где его
границы.

Категория должна быть одной из:

- `runtime`;
- `application`;
- `infrastructure`;
- `data`;
- `quality`;
- `documentation`;
- `developer-experience`;
- `browser`.

Пиши описание так, чтобы его понял человек, который впервые видит название.
Одна запись может быть библиотекой, host tool или сервисом, но всегда должна
объяснять роль именно в Brai New.

Поле `relatedTools` содержит только `id` других записей manifest. Если прямой
связи нет, можно оставить массив пустым. Все эти сведения попадут на отдельную
страницу инструмента и в JSON для будущего сайта.

## 3. Внутренне синхронизируй производные материалы

Агент выполняет эти действия сам после изменения manifest. Их не нужно просить
у пользователя и не нужно превращать в отдельный ручной этап:

```bash
pnpm run stack:generate
pnpm run stack:check
pnpm run docs:check
```

После этого проверь страницу в `docs/stack/tools/`, категорию в
`docs/stack/by-category/` и запись в `docs/stack/catalog.json`. Generated-файлы
не редактируются вручную: если текст нужно изменить, меняй manifest и повторяй
генерацию.

Если факты установки или человеческое объяснение неполны, агент сначала
уточняет источник истины или фиксирует блокер. Нельзя оставлять установленный
инструмент только в `/home/mark/DEPLOYMENT.md` или только в package manifest:
поддерживаемый stack должен иметь и подробную reader-facing страницу.

## 4. Заверши обычные проверки

Для изменения package/tooling configuration запусти релевантные `format:check`,
lint, typecheck и тесты. Для host installation дополнительно обнови
`/home/mark/DEPLOYMENT.md` в той же операции; каталог стека ссылается на него,
но не копирует host state.
