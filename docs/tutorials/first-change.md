# Первое безопасное изменение в Brai New

За этот tutorial ты пройдёшь короткий цикл изменения: поймёшь границу
workspace-проекта, проверишь контракты, запустишь CI и обновишь документацию.
Команды не меняют production host.

## Что понадобится

- Node.js `>=22.22.3 <23` и pnpm `>=11.13.1 <12`;
- checkout `/srv/projects/brai-new`;
- возможность запускать локальные команды проекта.

## Шаг 1. Найди источник поведения

Начни с Memory Bank и карты репозитория:

```bash
sed -n '1,220p' memory-bank/README.md
sed -n '1,220p' docs/reference/repository-map.md
```

Выбери один каталог `apps/`, `services/`, `packages/` или `infrastructure/`.
Затем прочитай его README, исходники и тесты. Для access/security сначала
прочитай `AGENTS.md` и соответствующий OpenSpec.

## Шаг 2. Получи первый проверяемый результат

Запусти тесты выбранного workspace-проекта, например:

```bash
pnpm --filter @brai/brai-factory test
```

Если меняешь web, используй `pnpm --dir apps/web test`. Успешный тест показывает,
что ты находишься в правильной границе и можешь отличить локальное поведение от
архитектурного контракта.

## Шаг 3. Проверь весь workspace

После изменения выполни:

```bash
pnpm run ci
```

Если менялись Compose или инфраструктурные файлы, добавь:

```bash
pnpm run compose:config
```

## Шаг 4. Обнови правильный тип документа

- новая процедура → `docs/how-to/`;
- точный параметр/команда → `docs/reference/`;
- объяснение причины → `docs/explanation/`;
- принятое архитектурное решение → `docs/decisions/`;
- изменение технологии → `docs/stack/`.

Не записывай секреты и не дублируй нормативный requirement. Сошлись на
источнике истины.

## Что получилось

Теперь у изменения есть источник, тестовая проверка и найденное место в
документации. Следующая задача обычно начинается с
[локального how-to](../how-to/local-development.md) и
[карты команд](../reference/commands.md).
