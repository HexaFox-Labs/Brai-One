# ADR-0001: Структура документации по Diátaxis и docs-as-code

- **Статус:** `accepted`
- **Deciders:** Сергей Bright, Mark
- **Дата:** `2026-07-18` UTC
- **Область:** документация репозитория Brai New

## Контекст

В проекте одновременно существуют нормативные OpenSpec, README отдельных
компонентов, Memory Bank и подробные документы по access/isolation. Без общей
схемы читатель смешивает учебник, командный runbook, справочник и rationale.
Версии стека также легко начинают расходиться с package manifests.

Проектный стандарт уже описывает применение Diátaxis в
`docs/documentation-methodology.md`. Нужна структура каталогов, которая
превращает этот стандарт в ежедневную навигацию.

## Решение

Использовать три связанные практики:

1. **Diátaxis** — четыре пользовательских режима чтения:
   `tutorials/`, `how-to/`, `reference/`, `explanation/`.
2. **ADR** — каталог `decisions/` для долгоживущих решений и trade-off-ов.
3. **Docs-as-code** — Markdown в репозитории, ссылки на источники истины,
   проверка ссылок/форматирования и обновление документа рядом с изменением.

Технологический стек выделен в `stack/`, потому что это отдельный постоянный
справочник, а не объяснение конкретной функции. Шаблоны собраны в `templates/`.

## Границы ответственности

- `AGENTS.md` и OpenSpec имеют приоритет над этой документацией;
- код и тесты определяют фактическое поведение;
- package manifests и lockfile определяют версии;
- host registry определяет установленное окружение;
- docs объясняют, связывают и делают эти источники обнаруживаемыми.

## Последствия

Плюсы:

- путь читателя виден по типу задачи;
- объяснения не загрязняют точные справочники;
- решения и причины не теряются в истории чата;
- стек можно обновлять независимо от архитектурных объяснений;
- новые материалы получают одинаковый шаблон и ссылки.

Цена:

- один факт может требовать обновления источника и краткой страницы-индекса;
- нужно поддерживать ссылки и не создавать дубликаты;
- не всякий старый документ сразу соответствует новой раскладке.

## Альтернативы

- **Один большой `docs/README.md`:** проще старт, но он быстро смешивает
  аудитории и плохо масштабируется.
- **Только README рядом с кодом:** удобно владельцу пакета, но нет системной
  карты, общих процедур и архитектурного контекста.
- **Отдельный docs-site framework:** преждевременно, пока проект не требует
  сборки сайта; plain Markdown проще проверять и читать локально.

## Проверка

- `docs/README.md` ведёт в каждый основной раздел документации.
- Шаблоны, stack-каталог и methodology находятся в ожидаемых каталогах.
- Markdown и локальные ссылки проходят документационные smoke-checks.

## Связанные источники

- `docs/README.md`
- `docs/documentation-methodology.md`
- `AGENTS.md`
- `memory-bank/README.md`
