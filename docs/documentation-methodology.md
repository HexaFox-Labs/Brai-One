# Как писать документацию и спецификации в Brai New

> Проектный стандарт для людей и агентов, которые создают или изменяют
> документацию, OpenSpec-спецификации, проектные решения и инструкции.

**Методология:** Diátaxis с отдельными правилами совместимости с OpenSpec  
**Статус:** обязательный проектный стандарт  
**Дата фиксации:** 18 июля 2026 года  
**Для кого:** разработчики, технические авторы и агенты

---

## Коротко

Перед написанием документа сначала нужно понять, **зачем читатель его
откроет**:

| Потребность читателя                            | Тип документа | Главный вопрос                 |
| ----------------------------------------------- | ------------- | ------------------------------ |
| Научиться с нуля на работающем примере          | Tutorial      | «Можешь провести меня?»        |
| Выполнить конкретную задачу                     | How-to        | «Как мне сделать это?»         |
| Найти точный факт, параметр, контракт или норму | Reference     | «Что именно здесь определено?» |
| Понять причины, устройство и компромиссы        | Explanation   | «Почему это устроено так?»     |

Эти четыре типа образуют методологию
[Diátaxis](https://diataxis.fr/). Они отличаются не оформлением, а задачей
читателя. Если смешать их без явных границ, документ становится неудобным:
оператор ищет команду и вынужден читать историю решения, а новичок получает
таблицу параметров вместо обучения.

В Brai New Diátaxis применяется так:

- постоянный `openspec/specs/<capability>/spec.md` — нормативный **Reference**;
- `proposal.md` и `design.md` активного OpenSpec change — в основном
  **Explanation**;
- `tasks.md` — план выполнения, близкий к **How-to**, но не заменяющий
  пользовательскую инструкцию;
- документы в `docs/` могут быть Tutorial, How-to, Reference, Explanation или
  явно размеченной композицией;
- ADR Brai New хранятся в `docs/decisions/`, собираются Log4brains в тёмный
  static output и после checks автоматически публикуются на `adr.brai.one`;
  ADR фиксирует rationale и trade-offs, но не заменяет нормативный OpenSpec;
- корневой `README.md` служит входной страницей и ведёт к подробным материалам;
- Memory Bank хранит краткий рабочий контекст, а не заменяет документацию,
  OpenSpec, код или тесты.

> [!IMPORTANT]
> Diátaxis не меняет синтаксис OpenSpec. Нельзя ради методологии помещать
> учебные шаги, рассуждения или историю решения внутрь нормативного
> `spec.md`. Требование и проверяемые сценарии должны оставаться точными и
> однозначными.

## Зачем нужен единый стандарт

Документация в проекте обслуживает несколько разных задач:

- владелец продукта хочет понять, что система делает и где её границы;
- разработчик ищет точный контракт;
- оператор хочет выполнить действие и проверить результат;
- новый участник хочет получить первый рабочий результат;
- агенту нужно отличить действующее требование от старого обсуждения;
- ревьюеру нужно проверить, соответствует ли реализация заявленному
  поведению.

Один неструктурированный текст плохо решает все эти задачи сразу. Обычно
появляются четыре проблемы:

1. **Нормы смешиваются с мнениями.** Непонятно, что обязательно, а что было
   лишь рассмотренной идеей.
2. **Инструкции смешиваются с объяснением.** Нужная команда теряется среди
   архитектурных рассуждений.
3. **Архив принимают за текущее состояние.** Агент читает старый proposal и
   повторяет уже отменённое решение.
4. **Документ объявляют готовым без проверки.** Ссылки не открываются,
   команды не работают, а пример противоречит коду.

Этот стандарт задаёт одинаковый порядок работы для человека и агента:
сначала определить потребность читателя и источник истины, затем исследовать
реализацию, выбрать тип документа, написать материал и проверить его.

## Что является источником истины

При расхождении материалов действует порядок из корневых правил проекта:

1. текущие инструкции Сергея и `AGENTS.md`;
2. постоянные нормативные OpenSpec-спецификации;
3. исходный код, конфигурация и тесты;
4. Memory Bank;
5. архивные `proposal.md`, `design.md`, `tasks.md` и другие материалы change.

Для документации из `docs/` источник зависит от утверждения:

| Утверждение в документе                  | Что нужно проверить                       |
| ---------------------------------------- | ----------------------------------------- |
| Обязательное поведение системы           | постоянную OpenSpec-спецификацию          |
| Фактическое поведение реализации         | код и тесты                               |
| Команду эксплуатации                     | действующий скрипт, unit или README       |
| Путь, порт, service unit, имя переменной | конфигурацию и установленное окружение    |
| Причину архитектурного решения           | design, ADR, комментарии и историю change |
| Текущий этап работы                      | `memory-bank/activeContext.md`            |

Нельзя превращать предположение в факт только потому, что оно звучит
правдоподобно. Если подтверждения нет:

- исследовать код, тесты и конфигурацию;
- явно отметить неизвестность;
- либо не включать утверждение.

### Архив OpenSpec

Архивный change объясняет, **как проект пришёл к решению**, но не определяет
текущий контракт. После завершения change:

- итоговые требования находятся в `openspec/specs/`;
- исходный change находится в `openspec/changes/archive/`;
- архив нельзя редактировать так, будто change снова активен;
- новое изменение оформляется новым Change или task-context и затем
  синхронизируется с постоянной спецификацией, если изменился нормативный
  контракт.

## Карта Diátaxis

Diátaxis делит документацию по двум осям:

- читатель **учится** или уже **работает**;
- читателю нужно **действовать** или **понимать**.

|                           | Действие                         | Знание                               |
| ------------------------- | -------------------------------- | ------------------------------------ |
| Читатель учится           | Tutorial                         | Explanation                          |
| Читатель выполняет работу | How-to                           | Reference                            |
| Основная форма            | управляемые шаги                 | точное описание или рассуждение      |
| Критерий успеха           | читатель получил навык/результат | читатель нашёл факт/понял устройство |

### Tutorial

Tutorial — учебный маршрут. Он проводит новичка от известной стартовой точки
к первому работающему результату.

Пример для Brai:

> «Создаём первый database-owning service и проводим запрос через NATS».

Tutorial не должен пытаться перечислить все параметры или доказать каждое
архитектурное решение. Для этого он ссылается на Reference и Explanation.

### How-to

How-to — инструкция для конкретной реальной задачи. Читатель уже знает
основы и хочет получить результат.

Пример:

> «Как проверить host-level preflight пользовательских sandbox».

How-to начинается с результата, содержит предусловия, точные шаги, проверку и
устранение типичных ошибок.

### Reference

Reference — точное и полное описание контракта или интерфейса.

Примеры:

- OpenSpec requirements и scenarios;
- HTTP/NATS contracts;
- таблица переменных окружения;
- перечень systemd units и путей;
- значения по умолчанию и ограничения.

Reference должен быть удобен для поиска. Он не должен превращаться в эссе.

### Explanation

Explanation раскрывает причины, модель, альтернативы и компромиссы.

Примеры:

- почему обычный пользователь получает одну постоянную среду;
- почему Gateway не получает database credentials;
- почему смена `developer_mode` заменяет runtime;
- почему квота является лимитом, а не резервированием места.

Explanation помогает понять решение, но не создаёт новую норму вместо
OpenSpec.

## Как выбрать тип документа

Начните с одного вопроса:

> Что читатель должен получить сразу после чтения?

Используйте дерево решения:

```text
Читатель должен выполнить действия?
├── Нет
│   ├── Нужен точный факт или полный контракт? → Reference
│   └── Нужно понять причины и устройство?     → Explanation
└── Да
    ├── Он учится и нуждается в сопровождении? → Tutorial
    └── Он уже работает и преследует цель?      → How-to
```

### Быстрая проверка заголовка

| Если заголовок похож на…                     | Вероятный тип |
| -------------------------------------------- | ------------- |
| «Создаём первый сервис Brai»                 | Tutorial      |
| «Как добавить миграцию сервиса»              | How-to        |
| «Конфигурация `brai-access`»                 | Reference     |
| «Почему межсервисный трафик идёт через NATS» | Explanation   |

### Не нужно создавать все четыре типа автоматически

Diátaxis — не требование завести четыре пустых файла для каждой функции.
Нужно создавать только те материалы, которые отвечают реальным потребностям.

Ориентир:

| Объект                   | Tutorial | How-to | Reference | Explanation          |
| ------------------------ | -------- | ------ | --------- | -------------------- |
| Пользовательская функция | часто    | да     | да        | при сложном решении  |
| API или NATS contract    | редко    | иногда | да        | редко                |
| Внутренняя архитектура   | нет      | редко  | да        | да                   |
| Конфигурация             | нет      | да     | да        | редко                |
| Процесс эксплуатации     | иногда   | да     | иногда    | при сложных причинах |
| Архитектурный принцип    | нет      | нет    | иногда    | да                   |

## Как Diátaxis связана с OpenSpec

OpenSpec и Diátaxis решают разные задачи:

- OpenSpec фиксирует изменение и нормативное поведение;
- Diátaxis организует знания под потребности читателя.

Они дополняют друг друга.

### Соответствие артефактов

| Артефакт                                      | Роль в проекте                                        | Ближайший тип Diátaxis  |
| --------------------------------------------- | ----------------------------------------------------- | ----------------------- |
| `openspec/specs/<capability>/spec.md`         | постоянный нормативный контракт                       | Reference               |
| `openspec/changes/<change>/proposal.md`       | проблема, цель, охват и границы изменения             | Explanation             |
| `openspec/changes/<change>/design.md`         | устройство, решения, альтернативы и компромиссы       | Explanation + Reference |
| `openspec/changes/<change>/tasks.md`          | проверяемый план реализации                           | рабочий How-to          |
| `openspec/changes/<change>/specs/.../spec.md` | delta требований, которую change добавляет или меняет | Reference               |
| `docs/tutorial-*.md`                          | безопасное обучение на работающем примере             | Tutorial                |
| `docs/how-to-*.md`                            | операторская или разработческая инструкция            | How-to                  |
| `docs/reference-*.md`                         | удобный технический справочник                        | Reference               |
| `docs/explanation-*.md`                       | понятное объяснение архитектуры                       | Explanation             |
| `README.md`                                   | карта входа и ссылки                                  | навигация               |
| `memory-bank/*.md`                            | краткий рабочий контекст агента                       | не Diátaxis-документ    |

### Нормативный `spec.md`

Постоянный `spec.md` отвечает только на вопрос:

> Какое поведение система обязана обеспечивать?

Он должен содержать:

- краткий `Purpose`;
- атомарные requirements;
- однозначные нормативные слова `SHALL`, `MUST`, `MUST NOT`, `MAY`;
- проверяемые scenarios в форме `WHEN` / `THEN` / `AND`;
- граничные и отказные сценарии, если они влияют на контракт.

В нём не должно быть:

- пошаговой инструкции по реализации;
- дневника работы агента;
- списка файлов, которые случайно изменились;
- неподтверждённых будущих идей;
- длинного сравнения альтернатив;
- утверждения «реализовано», если это не проверено;
- закрытых задач вместо требований.

### `proposal.md`

Proposal отвечает:

- какая проблема существует;
- почему изменение нужно сейчас;
- что войдёт в change;
- что явно не войдёт;
- какой пользовательский или системный результат ожидается.

Proposal не обязан описывать точную реализацию. Он фиксирует намерение и
границы.

### `design.md`

Design объясняет:

- доверительные границы;
- поток данных и управления;
- ключевые структуры и состояния;
- выбранные решения;
- альтернативы;
- компромиссы;
- отказные режимы;
- миграцию и rollout, если они нужны.

Каждое важное решение должно быть связано с конкретной проблемой. Формула:

> Выбрано X вместо Y, потому что Z. Цена выбора — Q.

### `tasks.md`

Tasks — не отчёт задним числом и не список общих пожеланий. Каждая задача
должна быть:

- ограниченной;
- проверяемой;
- связанной с требованием или design-решением;
- отмеченной выполненной только после фактической реализации и проверки.

Отдельно фиксируются:

- реализация;
- тесты;
- документация;
- инфраструктурная установка;
- acceptance;
- синхронизация постоянной спецификации;
- архивирование change.

### Завершение рабочей задачи

Рабочая задача считается завершённой только когда:

1. все обязательные tasks выполнены;
2. код и инфраструктура приведены в требуемое состояние;
3. проверки действительно прошли;
4. постоянная OpenSpec-спецификация синхронизирована, если изменился
   нормативный контракт, либо явно записано, почему она не менялась;
5. reader-facing документация отражает подтверждённое текущее поведение,
   если задача её затрагивает;
6. ADR-impact получил результат: ADR создан, обновлён, заменён или не нужен с
   явной причиной;
7. Memory Bank отражает новый текущий этап;
8. если работа велась через OpenSpec Change, Change заархивирован после всех
   перечисленных проверок.

Нельзя объявлять задачу завершённой только потому, что написаны proposal,
design или часть кода. Статусы `pending-governance`, `blocked`, `spec-drift`,
`conflict` и неподтверждённые claims оставляют задачу открытой.

## Обязательный процесс для агента

### Шаг 1. Прочитать правила и контекст

Перед работой агент читает:

1. `AGENTS.md`;
2. `memory-bank/README.md`, затем компактные `activeContext.md` и
   `progress.md`;
3. только тематические Memory Bank-файлы и источники, которые нужны выбранному
   маршруту;
4. постоянные OpenSpec-спецификации и активный Change, если они относятся к
   задаче;
5. существующую документацию, код, тесты и конфигурацию по затронутым
   поверхностям.

Маршрут (`OpenSpec Change`, task database или прямое изменение) и глубина
`quick`/`normal`/`full` определяются автоматически через `docflow`. Если
контекст неоднозначен, чтение и проверка углубляются максимум на один уровень;
это не является основанием автоматически создавать ADR или новую норму.

### Шаг 2. Зафиксировать аудиторию и результат

До написания нужно одной фразой сформулировать:

```text
После чтения [кто] сможет [получить проверяемый результат].
```

Примеры:

```text
После чтения оператор сможет проверить, что sandbox runtime готов к запуску.
После чтения разработчик сможет найти все поля launch contract и их ограничения.
После чтения владелец продукта поймёт, почему квота не резервирует диск.
```

Если в одной фразе одновременно появляются «научится», «выполнит», «найдёт
все параметры» и «поймёт причины», вероятно, нужны отдельные документы или
явные секции.

### Шаг 3. Исследовать предмет до написания

Агент обязан прочитать:

- реализацию целевой функции или компонента целиком;
- тесты, включая ошибки и граничные случаи;
- публичные contracts и schemas;
- конфигурацию и значения по умолчанию;
- связанные operator README;
- design/ADR, если утверждаются причины выбора.

Для большой темы полезно составить внутреннюю карту:

```text
Цель:
Аудитория:
Тип Diátaxis:
Источники истины:
Публичная поверхность:
Ключевые понятия:
Предусловия:
Отказные режимы:
Решения и компромиссы:
Команды и примеры, которые нужно проверить:
```

### Шаг 4. Составить карту доказательств

Каждое сильное утверждение должно иметь проверяемую опору:

| Утверждение | Источник       | Как проверено      |
| ----------- | -------------- | ------------------ |
| пример      | файл/тест/unit | команда или чтение |

Эту таблицу не обязательно публиковать. Она нужна автору, чтобы не
фантазировать.

### Шаг 5. Выбрать необходимые типы

Агент явно определяет, что будет создано:

```text
Tutorial: нужен / не нужен
How-to: нужен / не нужен
Reference: нужен / не нужен
Explanation: нужен / не нужен
```

Для OpenSpec change это решение применяется к сопровождающей документации.
Синтаксис самого OpenSpec остаётся нормативным.

### Шаг 6. Сначала написать Reference

Если работа включает Reference, он пишется первым. Это фиксирует словарь,
контракты, параметры и ограничения, на которые затем ссылаются другие
материалы.

### Шаг 7. Написать Explanation

После точных фактов можно объяснить:

- проблему;
- модель;
- причины;
- альтернативы;
- компромиссы;
- границы применимости.

Explanation ссылается на Reference и не дублирует весь контракт.

### Шаг 8. Написать How-to

How-to строится вокруг результата:

1. предусловия;
2. конкретные действия;
3. ожидаемый результат;
4. проверка;
5. устранение типичных ошибок;
6. откат, если действие меняет состояние и откат уместен.

### Шаг 9. Написать Tutorial, если он нужен

Tutorial создаётся только для реального пути обучения. Первый видимый
результат должен появиться не позднее третьего шага.

Tutorial не должен использовать production secrets, небезопасные фиктивные
права или обходить реальную архитектуру.

### Шаг 10. Связать документы

Новый материал должен:

- быть доступен из `README.md` максимум за два перехода;
- ссылаться на нормативную спецификацию, если описывает контракт;
- ссылаться на How-to или Reference из Tutorial;
- ссылаться на Reference из Explanation;
- не ссылаться на архив как на текущую норму.

### Шаг 11. Проверить

Минимальный набор:

```bash
pnpm exec prettier --check README.md AGENTS.md docs/*.md memory-bank/*.md
rg -n '<<<<<<<|=======|>>>>>>>' README.md AGENTS.md docs memory-bank openspec
```

Дополнительно по содержанию:

- проверить все относительные ссылки;
- выполнить безопасные команды из примеров;
- проверить Mermaid/Kroki-диаграммы рендерингом;
- проверить отсутствие секретов и реальных credentials;
- сверить имена, пути, порты, units и defaults с источником;
- запустить релевантные тесты, если документация утверждает поведение кода.

### Шаг 12. Обновить текущий контекст

После значимого изменения:

- обновить `memory-bank/activeContext.md`;
- обновить `memory-bank/progress.md`, если изменился этап или результат
  проверки;
- не копировать туда весь новый документ;
- оставить краткое резюме и ссылку.

## Правила для каждого типа документа

### Tutorial: правила качества

Tutorial должен:

- начинаться с ясного результата;
- указывать безопасную стартовую точку;
- показывать точные действия;
- давать видимый результат максимум к третьему шагу;
- объяснять новый термин при первом появлении;
- вести по одному рекомендуемому пути;
- завершаться итогом «что получилось» и следующими ссылками.

Tutorial не должен:

- перечислять все возможные варианты;
- превращаться в API reference;
- предлагать читателю самостоятельно выбрать критическую архитектуру;
- использовать шаг, который автор не проверил.

### How-to: правила качества

How-to должен:

- решать одну конкретную задачу;
- начинаться с глагольной цели;
- перечислять точные предусловия;
- содержать исполняемые команды или точные действия;
- показывать способ проверки;
- содержать troubleshooting, если операция может завершиться ошибкой;
- называть влияние и откат для опасных изменений.

How-to не должен:

- обучать предмету с нуля;
- отвлекаться на длинную теорию;
- скрывать важное условие в примечании после команды;
- утверждать успех по одному `active` unit без проверки реальной готовности.

### Reference: правила качества

Reference должен:

- быть полным в заявленном охвате;
- использовать стабильную структуру;
- перечислять типы, значения по умолчанию и ограничения;
- отделять обязательное от необязательного;
- содержать точные имена;
- быть удобным для поиска;
- ссылаться на источник истины.

Reference не должен:

- объяснять историю каждого решения;
- содержать непроверенные советы;
- смешивать текущие и будущие поля;
- копировать секреты или значения из закрытых файлов.

### Explanation: правила качества

Explanation должен:

- начинаться с проблемы;
- объяснять модель простыми словами;
- показывать причинно-следственные связи;
- называть альтернативы и цену выбора;
- отделять факт от вывода;
- ссылаться на Reference;
- честно указывать ограничения.

Explanation не должен:

- вводить новую обязательную норму только повествовательным текстом;
- маскировать решение словами «очевидно» или «просто»;
- подменять причины перечнем компонентов;
- скрывать компромиссы.

## Когда допустим один составной документ

Иногда Сергей явно просит один подробный Markdown-документ для простого
человека. В таком случае допустимо объединить несколько типов Diátaxis, если:

1. в начале есть короткое понятное Explanation;
2. учебные или операторские действия находятся в отдельных секциях;
3. точный Reference вынесен в отдельный справочный раздел;
4. содержание и якоря позволяют перейти сразу к нужному типу материала;
5. нормативный OpenSpec остаётся отдельным источником истины;
6. документ прямо сообщает, где объяснение, где инструкция, а где справочник.

Пример такого материала:
[`permissions-and-isolation.md`](permissions-and-isolation.md). Он объединяет:

- Explanation для владельца продукта;
- Reference по путям, units и ограничениям;
- How-to по проверке состояния;
- onboarding-элементы для первого знакомства.

Это осознанная композиция, а не разрешение перемешивать разные задачи в
каждом абзаце.

## Язык, названия и оформление

### Язык

- Reader-facing документация для Сергея и операторов пишется по-русски.
- Имена полей, API, файлов, команд и нормативные слова не переводятся.
- Термин при первом появлении кратко объясняется простыми словами.
- Английский допустим в коде, contracts и там, где перевод ухудшит точность.

### Названия файлов

Предпочтительные шаблоны:

```text
docs/tutorial-<topic>.md
docs/how-to-<task>.md
docs/reference-<component>.md
docs/explanation-<decision>.md
docs/<topic>.md
```

Последний вариант используется для крупного составного руководства с ясной
внутренней структурой.

### Заголовки

- Tutorial называет результат обучения.
- How-to начинается с «Как».
- Reference называет компонент или контракт.
- Explanation начинается с «Почему» или формулирует архитектурный вопрос.
- Заголовок не должен обещать больше, чем документ действительно покрывает.

### Команды

- Команда должна быть точной и копируемой.
- Нельзя использовать реальные секреты.
- Опасная команда требует описания влияния и безопасного контекста.
- Ожидаемый результат показывается рядом, если он неочевиден.
- Если команда зависит от текущего каталога, каталог указывается.

### Диаграммы

- Диаграмма нужна только когда связь, поток или граница понятнее визуально.
- Исходник диаграммы хранится как текст.
- Для Mermaid и других поддерживаемых форматов проверять рендеринг через
  локальный Kroki.
- Диаграмма не заменяет текстовое описание ключевых ограничений.

### Ссылки

- Использовать относительные ссылки для файлов репозитория.
- Ссылаться на постоянную спецификацию, а не на архивную копию.
- Внешняя ссылка должна вести на первичный источник.
- Новый документ должен быть достижим из корневого README максимум за два
  перехода.

## Проверка качества

### 1. Точность

- [ ] Каждое обязательное поведение совпадает с постоянным OpenSpec.
- [ ] Каждое утверждение о реализации совпадает с кодом и тестами.
- [ ] Имена файлов, services, units, переменных и портов точны.
- [ ] Примеры не используют удалённые или выдуманные интерфейсы.
- [ ] Будущее состояние не описано как уже реализованное.

### 2. Разделение типов

- [ ] Понятно, к какому типу Diátaxis относится документ или секция.
- [ ] Tutorial учит, а не перечисляет API.
- [ ] How-to ведёт к цели, а не читает лекцию.
- [ ] Reference описывает, а не убеждает.
- [ ] Explanation раскрывает причины, но не подменяет норму.

### 3. Полнота

- [ ] Заявленный охват покрыт целиком.
- [ ] Существенные граничные и отказные случаи названы.
- [ ] У инструкции есть предусловия и проверка.
- [ ] У архитектурного решения названы компромиссы.
- [ ] У OpenSpec change явно указаны границы и non-goals.

### 4. Понятность

- [ ] Первый экран объясняет назначение документа.
- [ ] Новые термины объяснены при первом появлении.
- [ ] Абзацы короткие, предложения конкретные.
- [ ] Читателю не нужно угадывать следующий шаг.
- [ ] Содержание и заголовки позволяют быстро найти нужный раздел.

### 5. Проверяемость

- [ ] Все относительные ссылки существуют.
- [ ] Форматирование Prettier проходит.
- [ ] Диаграммы рендерятся.
- [ ] Безопасные команды из примеров проверены.
- [ ] Утверждения о тестах содержат фактический результат, а не намерение.
- [ ] В документе нет секретов, credentials или содержимого закрытых env-файлов.

### 6. Согласованность проекта

- [ ] README ведёт к новому документу.
- [ ] `AGENTS.md` содержит относящееся к работе правило.
- [ ] OpenSpec синхронизирован, если изменился контракт.

## Типичные ошибки

### Ошибка: четыре пустых каталога на каждую тему

Diátaxis не требует создавать Tutorial, How-to, Reference и Explanation, если
для них нет потребности. Пустая структура увеличивает шум.

**Правильно:** выбрать только нужные типы и связать их.

### Ошибка: причины решения внутри normative requirement

Длинное объяснение внутри requirement затрудняет проверку и создаёт
двусмысленность.

**Правильно:** оставить обязательство и scenarios в `spec.md`, а причины
перенести в `design.md` или Explanation.

### Ошибка: tasks объявлены выполненными по факту написания плана

Чекбокс означает проверенный результат, а не намерение агента.

**Правильно:** закрывать задачу после изменения, теста и требуемого
acceptance.

### Ошибка: архивный change используется как текущая норма

Архив сохраняет контекст прошлого изменения и может содержать промежуточные
решения.

**Правильно:** использовать `openspec/specs/` для действующих требований.

### Ошибка: инструкция без проверки

После последнего шага читатель не понимает, всё ли сработало.

**Правильно:** добавить отдельный раздел «Проверка» с наблюдаемым результатом.

### Ошибка: справочник превращён в рассказ

Точные параметры трудно найти среди причин и истории.

**Правильно:** таблицы и стабильные разделы в Reference, причины по ссылке в
Explanation.

### Ошибка: документ описывает желаемое как установленное

Фраза «сервис работает» без runtime-проверки вводит следующего агента в
заблуждение.

**Правильно:** различать «спроектировано», «реализовано в коде», «проверено
тестами», «установлено на host» и «проверено в production».

### Ошибка: фантазирование недостающих компонентов

Нельзя придумывать broker, отдельного агента, сервис или роль ради красивой
схемы.

**Правильно:** каждый компонент на диаграмме должен иметь источник в коде,
конфигурации или утверждённой спецификации.

### Ошибка: документация существует, но её нельзя найти

Файл в `docs/` без ссылки из README фактически скрыт.

**Правильно:** добавить его в входную навигацию и связать с соседними
материалами.

## Как поддерживать документы после изменений

### Если меняется поведение

В выбранном рабочем контексте (`OpenSpec Change`, task database или прямой
зафиксированный task-context) нужно:

1. определить, изменилось ли нормативное поведение, текущее поведение или
   только текст;
2. обновить код и тесты;
3. синхронизировать постоянный `openspec/specs/.../spec.md` только при
   изменении будущего обязательного поведения; иначе записать причину
   `not-required`;
4. обновить reader-facing Reference/Explanation/How-to только там, где
   изменилось подтверждённое текущее состояние или пользовательский путь;
5. проверить ссылки и примеры;
6. оценить ADR impact и создать, обновить, заменить ADR или записать причину
   отсутствия ADR;
7. обновить Memory Bank;
8. если существует OpenSpec Change, архивировать его только после
   `pnpm run adr:check` и остальных маршрутизированных проверок.

### Если меняется только инструкция

OpenSpec не нужно менять, если нормативное поведение осталось прежним.
Необходимо:

- проверить актуальный интерфейс;
- обновить How-to или Tutorial;
- выполнить команды проверки;
- обновить ссылки и Memory Bank при значимой работе.

### Если меняется только объяснение

Нужно убедиться, что Explanation не вводит новую норму. Если новая мысль
меняет обязательное поведение, сначала требуется OpenSpec change.

### Если обнаружено расхождение

Не исправлять документ по памяти. Сначала:

1. определить тип утверждения;
2. найти более приоритетный источник;
3. проверить фактическое состояние;
4. исправить устаревший материал;
5. записать проверку в active context или progress.

## Готовая памятка для агента

Перед началом:

```text
[ ] Выполнил `docflow preflight` и определил route и глубину проверки.
[ ] Выбрал OpenSpec Change, task database или прямой task-context.
[ ] Определил аудиторию и результат чтения.
[ ] Выбрал Tutorial / How-to / Reference / Explanation.
[ ] Прочитал код, тесты, contracts и конфигурацию по теме.
```

Во время написания:

```text
[ ] Не смешиваю нормы, инструкции и причины без явных секций.
[ ] Не придумываю компоненты или факты.
[ ] Отделяю реализованное от запланированного.
[ ] Даю точные команды, пути и ограничения.
[ ] Ссылаюсь на источник истины.
```

Перед завершением:

```text
[ ] Проверил ссылки.
[ ] Проверил форматирование.
[ ] Проверил команды и диаграммы.
[ ] Проверил отсутствие секретов.
[ ] Добавил входную ссылку из README.
[ ] Синхронизировал OpenSpec, если менялся контракт.
[ ] Обновил activeContext.md и при необходимости progress.md.
[ ] Не объявил задачу завершённой до закрытия governance; Change архивируется,
    только если он действительно использовался.
```
