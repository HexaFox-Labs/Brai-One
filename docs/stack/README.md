# Технологический стек

**Статус:** `active`  
**Дата снимка:** `2026-07-18` UTC  
**Источник версий:** package manifests, `pnpm-workspace.yaml` и
`pnpm-lock.yaml`

Этот каталог отвечает на вопрос «что мы используем и зачем». Номера версий
здесь являются удобным индексом. При расхождении с манифестом приоритет имеет
манифест, после чего этот каталог обновляется.

Полные человеко-понятные mini-landing pages находятся в автоматически собранном
[каталоге инструментов](catalog.md). Его исходник —
[`tools/stack/catalog.json`](../../tools/stack/catalog.json); отдельные страницы,
категории и JSON для будущего сайта нельзя редактировать вручную.

## Как читать записи

Каждая страница инструмента подробно объясняет:

- что это такое и зачем оно нужно Brai;
- почему выбран именно этот инструмент и как он работает в нашем контуре;
- возможности, практические сценарии и типичные ошибки;
- где он находится, какие у него ограничения, статус и версия;
- как проверить установку и где читать дальше;
- какие инструменты связаны с ним.

Не добавляй библиотеку в этот каталог только потому, что она существует в
транзитивном `node_modules`. Указывай прямую зависимость workspace-проекта или
явно обозначай инфраструктурный инструмент.

## Политика выбора

- Прикладной трафик между Gateway, сервисами и workers идёт через NATS.
- Web и Gateway не получают database credentials.
- Сервисы-владельцы данных используют собственную Supabase schema и least
  privilege role.
- Приложения слушают loopback, публичный вход идёт через Caddy на 80/443.
- Новый инструмент сначала должен иметь владельца, назначение, способ проверки
  и место установки. Host tooling размещается под `/srv/opt/`, а не в checkout.
- Версии обновляются через манифест и lockfile; ручное изменение только этой
  страницы запрещено.

## Жизненный цикл стека

1. Идея или новый инструмент появляется в ADR либо в OpenSpec change, если он
   меняет контракт или security boundary. ADR Brai New создаётся в
   `docs/decisions/` и проверяется `pnpm run adr:check`.
2. После принятия он устанавливается под `/srv/opt/` или добавляется в
   project manifest и проходит CI или host acceptance.
3. В той же операции агент добавляет запись в
   [`tools/stack/catalog.json`](../../tools/stack/catalog.json) с категорией,
   подробным объяснением «что это» и «зачем нужно», назначением, типом,
   версией, источником и проверкой.
4. Агент внутренне запускает `stack:generate` и `stack:check`; они обновляют
   страницы, категории и web-ready JSON. Пользователь не должен отдельно
   инициировать эту синхронизацию.
5. При удалении инструмент помечается как `deprecated` на время перехода, а
   затем удаляется из каталога после удаления из кода и окружения.
