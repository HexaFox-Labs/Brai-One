# Design

## Context

Проект уже содержит нормативные OpenSpec-спецификации, Memory Bank и
установленную `@fission-ai/openspec` CLI. OpenSpec создаёт project-local skills
и Codex prompts, но их наличие само по себе не заставляет агента выбирать
OpenSpec workflow для обычной естественно-языковой задачи.

## Trust and system boundaries

- Пользовательское сообщение задаёт намерение и ожидаемый результат, но не
  меняет обязательные проектные ограничения.
- `AGENTS.md` задаёт operational policy агента.
- Постоянные `openspec/specs/*/spec.md` задают нормативное поведение.
- Код, конфигурация и тесты подтверждают фактическое состояние.
- `memory-bank/` хранит краткий контекст и не заменяет нормативные источники.

## Data and control flow

1. Агент читает правила, Memory Bank, релевантные specs и активные changes.
2. Агент классифицирует задачу как тривиальную или требующую durable change.
3. Для durable change агент продолжает совпадающий active change либо создаёт
   новый change и заполняет proposal, delta-spec, design и tasks.
4. Агент реализует tasks, обновляя их по фактическому прогрессу.
5. Агент запускает релевантные проверки и OpenSpec validation.
6. После успешной проверки агент синхронизирует постоянные specs и архивирует
   change; planning-only запрос оставляет change активным.

## Decisions

### Естественный язык — основной интерфейс

Выбрано автоматическое определение workflow вместо требования `/opsx:*`, потому
что пользователь взаимодействует с агентом через обычные команды и не обязан
знать внутренний CLI. Цена выбора — агент обязан явно классифицировать scope и
отчитываться о выбранном change.

### Два слоя правил

Operational policy находится в `AGENTS.md`, а контекст, который OpenSpec
вкладывает в сгенерированные инструкции, — в `openspec/config.yaml`. Это
снижает риск, что обновление CLI удалит проектные правила из generated files.

### Тривиальные задачи не получают лишнюю церемонию

Одноразовые read-only проверки, форматирование, исправление опечатки и
тест-only изменения без смены поведения могут выполняться без change. Любое
изменение API, NATS-контракта, схемы данных, доступа, инфраструктуры,
архитектуры, поведения или нескольких связанных компонентов требует change.

## Failure modes

- Нет совпадающего active change → агент создаёт новый, а не продолжает
  архивный.
- Несколько совпадающих changes → агент выбирает только при ясной ссылке;
  иначе просит уточнение.
- CLI или generated skill недоступны → агент эмулирует workflow, создавая
  артефакты вручную, и сообщает ограничение.
- Validation или тесты не проходят → change не архивируется и не объявляется
  завершённым.
- Пользователь просит только план → код не изменяется, change остаётся active.

## Rollout and rollback

Правила включаются добавлением `AGENTS.md` и `context` в
`openspec/config.yaml`. Generated Codex skills/prompts пересобираются через
`openspec update`. Откат возможен удалением добавленных policy-инструкций и
возвратом project context; нормативные спецификации не переписываются задним
числом.
