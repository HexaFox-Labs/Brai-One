# Автоматическая публикация ADR

## Почему

`docs/decisions/` уже является каноническим источником ADR Brai New, а
`adr.brai.one` уже умеет безопасно раздавать атомарно переключаемый static
release. Однако между изменением принятого ADR и обновлением сайта оставался
ручной publish-шаг. Это создаёт расхождение между источником решений и
читательским сайтом и заставляет Сергея участвовать в технической операции.

## Что меняется

- Добавляется root-owned systemd path/timer automation для ADR source.
- Сервис запускает сборку от непривилегированного `mark`, проверяет ADR,
  документацию и OpenSpec, а затем использует существующий атомарный publisher.
- Изменение valid ADR source после закрытия записи инициирует немедленную
  публикацию; минутный reconciliation timer страхует пропущенное событие.
- При ошибке проверок или сборки предыдущий `current` release остаётся активным.
- Состояние source manifest устраняет лишние публикации без изменения source.
- Static output принудительно получает тёмную тему Brai: один versioned CSS
  asset подключается ко всем HTML-страницам после сборки Log4brains.
- Исправляется timezone-дефект Log4brains: date-only ADR не может отображаться
  как следующий день; future date блокируется до публикации.
- `docs/decisions/` фиксируется как каталог принятых ADR: draft-материалы не
  добавляются в публикуемый source catalog.

## Границы

- ADR-site остаётся static publication, а не NATS-сервисом или контейнером.
- Новый внешний порт, database credentials, NATS, Caddy reload и изменения
  DNS не добавляются.
- Автоматизация не публикует другие документы или application releases.
- Тема не меняет исходники Log4brains, не требует клиентского переключателя и
  не добавляет runtime-сервис.
- CI/CD может позднее вызывать тот же publisher, но Git/remote workflow не
  является предусловием этой задачи.

## Затронутые capabilities

- `adr-publication` — добавляется автоматический безопасный trigger и
  fail-closed reconciliation.
