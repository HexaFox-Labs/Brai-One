# Brai agent access foundation

## Why

Brai должен безопасно запускать много параллельных пользовательских агентов и одновременно позволять явно доверенным пользователям развивать саму Brai. Старый проект смешивал разных Unix-писателей, runtime и административные DB-роли, preview-схемы, deploy-скрипты и ремонт прав. Это породило повторяющиеся `EACCES`, `SQLITE_READONLY`, schema drift, исчерпание соединений и stale pool failures.

Новая система получает две заранее заданные границы исполнения. Обычный пользователь работает только в своей постоянной среде. Пользователь, которому администратор включил режим разработчика, запускает веб-агента с теми же правами `mark + sudo`, что Codex Desktop.

## What changes

- Backend хранит администраторский флаг developer mode и до запуска детерминированно выбирает `user-sandbox` или `developer`.
- Обычный пользователь получает одну постоянную изолированную среду для всех своих агентов, проектов, контейнеров и локальных БД.
- Developer web-agent запускается на хосте как `mark` с действующим sudo-контрактом и прямым доступом к `/srv/projects/brai-new`.
- Пользовательская квота ограничивает фактическое потребление, ничего заранее не резервирует и включает все изменяемые данные пользователя.
- SQLite становится zero-config БД по умолчанию; пользовательский Postgres может работать только внутри пользовательской среды. Core Supabase остаётся только для сервисов Brai.
- Runtime/preflight-проверки закрепляют единственного штатного владельца checkout и запрещённые mount/credential boundaries.

## Boundaries

- Ни AI-модель, ни отдельный агент, ни prompt не участвуют в принятии решений о правах.
- Режим разработчика является полным доверием: пользователь с `mark + sudo` намеренно способен изменить весь хост. Это не sandbox.
- Нет полного Git clone или постоянной файловой копии на каждого агента или задачу. Агенты одного пользователя разделяют одну среду; общие immutable OCI layers хранятся один раз.
- Дисковый лимит не означает зарезервированное место и не вычитается из свободного объёма при создании пользователя.
- Пользовательские проекты не получают schemas, roles, admin DSN или service-role key основной Supabase Brai.
- Отдельного физического диска нет. На существующем разделе используется один общий ограниченный sparse XFS pool-файл; он не создаётся на пользователя или агента и не предвыделяет свой логический размер.
- GitHub, CI/CD, production deployment, multi-host sharding, managed Postgres и автоматическая публикация доменов не являются условиями завершения этого change. Их существующие заготовки не расширяют и не ослабляют access contract.
