# Дизайн

## Канонический контент

`tools/stack/catalog.json` остаётся единственным редактируемым источником.
У каждой записи `details` появляются `whatItIsDetailed` и
`whyNeededDetailed`. Оба поля должны быть непустыми многофразными текстами;
validator проверяет минимум два предложения в каждом поле и минимальную длину,
достаточную для содержательного объяснения.

Короткие `summary`, `whatItIs` и `purpose` сохраняются как индексные тексты,
но страница использует подробные поля. Это не ломает будущий web JSON и даёт
сайту короткий preview вместе с полным описанием.

## Автоматический install workflow

Корневой `AGENTS.md` фиксирует обязательный протокол для агента:

1. после фактической установки, обновления или удаления определить источник,
   версию, место, назначение и способ проверки;
2. обновить `/home/mark/DEPLOYMENT.md` для host-level изменения и package/config
   source of truth для project-level изменения;
3. сразу добавить или обновить запись в `tools/stack/catalog.json`, включая
   развёрнутые объяснения и классификацию;
4. внутренне вызвать генератор и проверки, не перекладывая команды на
   пользователя;
5. остановиться с ошибкой, если сведения неполны или generated output не
   синхронизирован.

Это agent workflow, а не новый пользовательский UI и не postinstall hook для
каждой транзитивной зависимости: только агент знает, является ли установленный
инструмент частью поддерживаемого Brai stack.

## Представление страницы

Разделы «Что это такое» и «Зачем это нужно Brai» получают сначала подробный
текст, затем существующие local operating model, capabilities, scenarios и
остальные operational sections.
